Welcome to this week's blog post, where we take a closer look at SQL Injection (SQLi). This vulnerability is a structural failure where a web application collapses the boundary between control code and user data, confusing untrusted input with executable logic. Why does this matter? A database is a rigid machine that blindly runs whatever text it receives, completely unaware of the developer's original intent. In a classic, in-band SQLi attack, the attacker exploits this blind trust by inserting malicious code directly into a standard input field—forcing the database to execute unauthorised commands and return the results immediately. This post breaks down the mechanics of this vulnerability, the mindset required to exploit it and the strategies to remediate it.

## The Core Mechanism: Context Confusion

SQLi occurs because of a fundamental flaw: context confusion.

Web applications often build database quires by sticking text strings together. When an application taes user input and injects it directly into a query string without safety checks, the database connot tell where the developer's instructions end and the uers's data begins. It treats the entire string as a single command.

## The Exploit Mindset: Breaking the Syntax

To exploit an in-band SQLi vulnerability, an attacker loos for input fields that feed directly into a database query (like login forms, search bars or URL parameters). The mindset focuses on breaking and reshaping the intended SQL syntax.

1. Injecting Control Characters: The attacker inputs characters like a single quote (') or a semicolon (;). This prematurely closes the develoer's data field.
2. Altering Logic: the attacer appends new SQL commands, such as `OR '1'='1'`. Because `1=1` is always true, this modifies the query logic to bypass authentication or expose hidden data.
3. Suppressing Errors: The attacer adds a comment symbol (like -- or # ) to tell the database to ignore the rest of the developer's original query, preventing syntax errors that could break the attack.

## The Remediation Strategy: Absolute Separation
Fixing SQLi requires enforcing a string, permanent boundary between code and data.
- User Parameterised Queries: Also known as prepared statements. This is the primary defense. It forces the database to cimpile the SQL code first, treating user input strictly as a literal value, never as executable code.
- Employ Object-Relational Mapping (ORM): Modern frameworks (like Entity Framework or Hibernate) naturally use parameterisation by default.
- Apply the Principle of Least Privilege: Restrict database permissions for the web application. If an app only needs to read data, deny it the rights to drop tables or access system configurations.
- Validate and Sanitise Input: Implement stric allow-lists for expected input formats, though this should only ever be used as a secondary layer of defense.

## Example Code: Vulnerable vs. Secure
### Scenario: A User Profile Search Bar
Imagine a web application with a search bar where users can look up profiles by entering a username.

1. The Vulnerable Code (String Concatenation)  
   This implementation creates the SQLi query by directly inserting the user's input into the command string.  
   
   ```python
   import sqlite3
   
   def get_user_vulnerable(user_input):
     # Connect to the database
     conn = sqlite3.connect("users.db")
     cursor = conn.cursor()
      
     # DANGER: Directly combining input with the query string
     query = f"SELECT * FROM profiles WHERE username = '{user_input}'"
      
     # Executes the combined string as a single command
     cursor.execute(query)
     return cursor.fetchall()
   ```  

   #### How the Attack Works  

   - Intended Input: If a user searches for `alice`, the query becomes: `SELECT * FROM profiles WHERE username = 'alice'` (Safe).
   - Malicious Input: An attacker enters: `alice' OR '1'='1`
   - The Resulting Query:

   ```sql
   SELECT * FROM profiles WHERE username = 'alice' OR '1'='1'
   ```

   - Why it Fails: The single quote in the input breaks out of the data field. Because `'1'='1'` is always ture, the database bypasses the username check entirely and returns every single profile in the database.  
  
  
2. The Secure Code (Paramaterised Queries)  
   This implementation uses paramterised queries (prepared statements). It passes the user input as a seperate argument, ensuring the database treats it strictly as a value.

   ```python
   import sqlite3

    def get_user_secure(user_input):
      # Connect to the database
      conn = sqlite3.connect("users.db")
      cursor = conn.cursor()
        
      # SECURE: The '?' placeholder marks where data will go
      query = "SELECT * FROM profiles WHERE username = ?"
      
      # Pass the input as a tuple in the second argument
      cursor.execute(query, (user_input,))
      return cursor.fetchall()
    ```  
   #### Why This is Safe
   - The Database Behavior: When the database looks at `query`, it compiles the structure *before* looing at the data. It pre-determines that the query will only ever search for a username.
   - Handling Malicious Input: If the attacer enters `alice' OR '1'='1`, the database literally searches or a user whose username matches that exact string: `alice' OR '1'='1`.
   - The Outcome: No syntax is broen, no logic is altered and the attack fails completely.
